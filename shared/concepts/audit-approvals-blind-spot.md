# Аудит approvals — слепая зона конфига

## Суть проблемы

При полном аудите конфига Hermes (май-июнь 2026) секция **`approvals`** ни разу не проверялась. Аудит шёл по жалобам Владимира, а жалоб на approval mode не было — и секция выпала.

## Хронология (из daily-бэкапов)

| Дата бэкапа | `approvals.mode` | Статус |
|---|---|---|
| 2026-06-01 04:00 | `manual` | ✅ Норма |
| 2026-06-02 04:00 | `false` (= off) | ❌ Без подтверждений |
| 2026-06-03 04:00 | `false` (= off) | ❌ Без подтверждений |
| 2026-06-04 04:00 | `false` (= off) | ❌ Без подтверждений |
| 2026-06-05 04:00 | `manual` | ✅ Вернулось |

## Что произошло

- С 2 по 4 июня `approvals.mode` было `false`, что код Hermes читал как `off` — полный bypass подтверждений
- 5 июня кто-то вернул обратно в `manual`
- Кто и когда изменил — в сессиях следов нет
- Скорее всего: ручная правка через `nano config.yaml` или `hermes config set approvals.mode off`

## Механизм

**Два независимых слоя защиты:**

1. **tirith** — внешний security scanner, сканирует команды на homoglyph/pipe-to-interpreter/heredoc. Включён всегда (`tirith_enabled: true`), срабатывает до approvals.
2. **approvals.mode** — окно пользователя. `manual` → спрашивает `/approve`, `smart` → LLM решает, `off` → не спрашивает.

## Решение

- Создан inotify-демон (`inotify_backup_daemon.sh`), который автоматически копирует config.yaml в `pre-edit/` при каждом изменении
- Навык `hermes-config-management` обновлён: включает чеклист аудита ВСЕХ секций, включая approvals, + процедуру проверки daily-бэкапов

## Правило на будущее

При любой жалобе на поведение подтверждений — проверять **НЕ ТОЛЬКО текущий конфиг**, но и **daily-бэкапы за 3-7 дней**. Текущий конфиг может быть уже исправлен, а бэкапы покажут реальную хронологию.
